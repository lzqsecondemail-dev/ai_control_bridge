# ACB 治理与未来工作线点子候选清单

> **创建时间**: 2026-06-03
> **说明**: 本文件记录 ACB 治理架构与未来产品路线的候选点子，均处于"挂起"状态。任何点子需经总控重新判断优先级后才可转为正式任务卡。

---

## 点子 1：Execution Checkpoint / Safe Step

### 背景
M3-UI.3-B.1 编码损坏 / 回滚事故暴露出当前 ACB 缺少强制 checkpoint 机制。事故后已新增治理规则要求每次任务完成后建立 checkpoint，但 checkpoint 管理目前依赖人工记忆和任务卡约束，未产品化。

### 核心思路
- ACB 未来应产品化"执行检查点"模块。
- 可视化展示：
  - patch backup 列表（local patch backup directory）
  - staged checkpoint 状态
  - checkpoint commit 列表（`git log` 中带 "Checkpoint" 前缀的提交）
  - 可恢复点列表（含时间戳、任务卡 ID、变更摘要）
  - diff 查看（checkpoint 间的变更对比）
- 一键恢复到指定 checkpoint（需二次确认）。
- 控制台顶部或 Git 面板中显示"最近 checkpoint 时间"和"checkpoint 覆盖状态"。

### 关键约束
- checkpoint 恢复是高危操作，必须经总控确认。
- 恢复前自动做新的 patch backup。
- 不得自动恢复。

### 状态：挂起

---

## 点子 2：Controlled Dispatch Bridge / 受控执行桥

### 背景
当前 ACB 的工作流止于"发送到 VS Code 查看端"（M3-Bridge.4）。用户仍需手动复制任务卡、切换到执行端（Codex / Claude Code / DeepSeek / PowerShell）、粘贴并执行。这本质上仍是手工转发。

### 核心思路
- ACB 终局不能只停留在复制任务卡。
- 长期目标是用户在 ACB 内显式点击确认后，将完整 bounded ACB_TASK_CARD 投递到对应执行端并触发执行。
- 核心原则是**"自动搬运，人工决策"**：
  - ACB 负责将任务卡搬运到正确的执行端；
  - 用户在 ACB 控制台显式点击确认后才触发搬运；
  - 搬运后执行端才开始执行；
  - 执行结果回报回 ACB。
- Dispatch 前必须通过完整门禁链：Payload Preflight → Execution Readiness Gate → Dispatch Gate。

### 关键约束
- 不得静默派发。
- 不得绕过 Payload Preflight / Readiness Gate。
- 不得在没有用户显式确认的情况下触发执行。
- 执行端识别（Codex / DeepSeek / Claude Code）必须准确，不得投错目标。

### 状态：挂起

---

## 点子 3：Quick Dispatch Badge / 快捷投递浮标

### 背景
当前用户需要打开 ACB 浮动控制台才能看到任务卡状态和发送入口。这在快速操作场景中不够便捷。

### 核心思路
- 当前 GPT 对话旁增加轻量入口（浮标 / badge / 小图标）。
- 只提示和触发快捷投递：
  - 显示当前对话是否有待发送任务卡；
  - 显示快捷投递目标（如 "发送到 DeepSeek"）；
  - 点击后进入确认弹窗，确认后触发搬运。
- 浮标是轻量入口，不做复杂交互。
- 浮标不得绕过 Payload Validation / Preflight / Readiness / Dispatch Gate。

### 关键约束
- 浮标是快捷入口，不是替代控制台。
- 不得绕过门禁链直接派发。
- 不得在浮标中展示完整任务卡内容（避免信息泄漏）。

### 状态：挂起

---

## 点子 4：Execution Report Return Bridge / 执行回报回传桥

### 背景
当前 ACB 的执行流程是单向的：从 GPT 对话捕获反馈 → 生成任务卡 → 发送到执行端。执行端完成任务后，回报仍需用户手动总结并带回总控对话。这破坏了闭环。

### 核心思路
- ACB 未来应识别执行端的回报：
  - Codex / DeepSeek / Claude Code 完成任务后的输出。
  - `git diff` / `git log` 变更摘要。
  - 测试结果（pass/fail 计数）。
- 自动生成"回总控草稿"：
  - 摘要执行结果；
  - 附带变更文件列表、commit diff、测试状态；
  - 生成草稿，不自动发送。
- 早期必须人工确认后才能将回报带回总控对话。

### 关键约束
- 生成草稿不等于自动发送。
- 回报内容必须人工审查确认。
- 不得自动上传执行结果到云端。

### 推荐技术路线
```
ACB → Local Bridge / VS Code Extension
  → local outbox task file（任务卡）
  → Codex CLI / DeepSeek / Claude Code 执行
  → local report file（执行回报）
  → ACB 生成回总控草稿
  → 用户确认后带回总控对话
```

### 状态：挂起

## 点子 5：Local Bridge + Outbox / Reports 文件队列

### 背景
当前 ACB 的任务卡发送和回报接收依赖浏览器 ↔ VS Code Bridge 的实时通信。如果执行端离线（如 Claude Code 未运行），任务卡会丢失。需要持久化的任务队列。

### 核心思路
- 建立本地文件队列：
  - `outbox/` — 待发送给执行端的任务卡（Markdown 文件）。
  - `reports/` — 执行端完成后写入的执行回报。
- 执行端（Codex CLI / DeepSeek / Claude Code）轮询或监听 `outbox/` 目录：
  - 发现新任务卡 → 读取 → 执行 → 写入回报到 `reports/`。
- ACB 轮询 `reports/` 目录，发现新回报后生成回总控草稿。

### 文件格式
- 任务卡：`task_{taskCardId}.md`，包含完整 bounded ACB_TASK_CARD。
- 回报：`report_{taskCardId}.md`，包含执行结果摘要、Git 状态、变更文件列表。

### 关键约束
- 文件队列是本地文件系统操作，不涉及网络。
- 不得自动执行 outbox 中的任务卡——执行端必须有人工确认或本地守护进程的受控逻辑。
- 不得将 reports 内容自动上传。

### 状态：挂起

## 点子 6：Executor Endpoint Research / 执行端接入调研

### 背景
ACB 未来需要支持多种执行端。当前仅验证了 VS Code 查看端（只读审核）。实际执行需要调研各执行端的接入方式。

### 待调研执行端

| 执行端 | 接入方式候选 | 优先级 |
|--------|-------------|--------|
| Codex CLI | `codex exec` 子命令 + 文件输入 | 高 |
| Codex App | GUI 端手动粘贴 vs CLI | 中 |
| Codex IDE Extension | VS Code 内联执行 | 中 |
| Claude Code | CLI 模式 + 文件输入 / stdin | 高 |
| DeepSeek | cc-switch / Zolt / WebChat paste | 中 |
| PowerShell | 本地脚本 + 文件输入 | 低 |
| 本地脚本 / Make / Just | 文件触发 | 低 |

### 调研输出
每个执行端应产出：
1. 最小可执行示例（发送一个简单任务卡并获得执行结果）
2. 约束与限制（命令行参数、token 限制、并发限制、超时）
3. 接入复杂度评估
4. 推荐接入方式

### 状态：挂起

## 点子 7：Branch Controller / 分控与可用分支

### 背景
当前 ACB 是单主线线性推进（master 分支）。未来若需要并行工作（如同时推进 M3-UI.3 和 M3-Bridge.5），当前模型不支持。

### 核心思路
- 总控未来可创建受控工作线：
  - 创建 branch + worktree；
  - 生成状态包（当前 commit、allowedFiles、checkpoint 策略、回报协议、合并目标）；
  - 指定分控（分支负责人）；
  - 分控在自己工作线内推进。
- 分控的职责：
  - 在分配给自己的工作线内推进开发；
  - 执行总控指定的任务；
  - 建立 checkpoint；
  - 回报进展。
- 分控不得：
  - 自行合并主线；
  - 跨工作线修改文件；
  - 修改其他分支的代码；
  - 自行创建子分支。

### 关键约束
- 合并回主线必须经总控审查和授权。
- 工作线状态必须可审查。
- 分控不是"自由 Agent"，而是"受控执行分支负责人"。

### 状态：挂起

## 点子 8：Team Mode / 客户领取受控分支

### 背景
ACB 个人版确认用户长期只需面对一个执行总控。但在团队场景下，可能需要多人协作——客户验收、顾问审查、执行端并行推进。

### 核心思路
- 团队模式下客户可向总控申请或领取受控分支工作线。
- 总控负责分配：
  - branch / worktree
  - 分控 / Agent 指定
  - 权限范围（allowedFiles、forbiddenActions）
  - 验收标准
  - 合并目标
- 客户不是自由创建裸 Git 分支。
- 客户在自己的工作线内推进，完成后请求合并。
- 总控审查后合并。

### 关键约束
- 团队模式是个人版的扩展，不是替代。
- 个人版保持单主线线性推进。
- 客户不能绕过总控直接派发执行端。
- 客户不能自行合并到主线。

### 状态：挂起

## 点子 9：总控四向对接模型

### 背景
随着 ACB 产品复杂化，总控需要与多类角色对接。当前文档中这些关系分散在各处，需要一次系统化梳理。

### 模型定义

总控（Controller）作为 ACB 项目的核心主脑，对接四类角色：

```
            ┌──────────────────────────────┐
            │          总控 (Controller)      │
            │  路线 · 派发 · 边界 · 审查 · 收口  │
            └────┬──────┬──────┬──────┬─────┘
                 │      │      │      │
        ┌────────┴─┐ ┌──┴───┐ ┌┴──────┴┐ ┌┴──────────┐
        │   客户    │ │ 顾问  │ │直属Agent│ │   分控     │
        │  需求     │ │ 审查  │ │ 执行    │ │  工作线    │
        │  验收     │ │ 建议  │ │ 回报    │ │  推进     │
        │  授权     │ │ 专业  │ │ 按卡    │ │  受控     │
        │  优先级   │ │ 判断  │ │ 操作    │ │  隔离     │
        └──────────┘ └──────┘ └─────────┘ └───────────┘
```

| 角色 | 职责 | 输入 | 输出 | 不得做 |
|------|------|------|------|--------|
| **客户** | 需求定义、验收确认、优先级判断、授权 | 产品需求、验收意见 | 需求描述、优先级、验收结论 | 直接派发执行端、自行合并代码 |
| **顾问** | 专业审查、风险提示、备选方案、建议 | 总控咨询 | 顾问建议、审查意见、任务草案 | 直接派发执行端、自行决定路线 |
| **直属 Agent** | 按任务卡执行明确任务 | ACB_TASK_CARD | 执行结果、Git 状态、回报 | 自行设计架构、扩大范围、跨 checkpoint 回滚 |
| **分控** | 受控工作线推进 | 工作线状态包 + 任务卡 | 进展汇报、checkpoint、合并请求 | 自行合并主线、跨工作线修改 |

### 产品意义
- 总控不是"超级执行者"，而是"路线收敛中枢"。
- 四向模型明确了信息流和权限边界，防止角色越界。
- 为后续分控、团队模式、多执行端并行提供架构基础。

### 状态：挂起

## 后续处理原则

1. 本文件只做点子沉淀，不自动转化为开发任务。
2. 任何挂起点子需经总控重新判断优先级后，才可生成正式 ACB_TASK_CARD。
3. 未来如有新挂起点子，可继续由执行端更新本文档。
4. 如需将某个点子提升为开发任务，应由总控生成带完整验收标准的任务卡，并在本文档对应点子上标注"已转为任务卡"及任务卡 ID。
5. 点子之间若有重叠或依赖关系，应在转化为任务卡前由总控梳理。
